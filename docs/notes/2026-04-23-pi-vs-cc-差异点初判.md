# 2026-04-23 pi vs cc 差异点初判

## 结论

`pi` 和 Claude Code 最大的差异点，不是能力多少，而是产品哲学反过来了。

- **Claude Code** 更像强成品 agent
- **pi** 更像可编程宿主

一句话说：

> cc 卖的是一个已经很强的官方 agent；pi 卖的是一套你可以继续长出自己工作流的 runtime host。

---

## 从源码看出来的第一层差异

## cc：内建工作流优先
关键锚点：
- `cc/src/tools/AgentTool/builtInAgents.ts`
- `cc/src/tools/AgentTool/loadAgentsDir.ts`
- `cc/src/tools/AgentTool/forkSubagent.ts`

当前判断：
- 官方内建 agent 是产品核心的一部分
- fork/subagent 是内建路径，不是示例扩展
- 官方 feature gate、cache-sharing、fork child 规则都被写进核心实现

这意味着：

> 高级工作流优先被做成产品内核能力。

## pi：宿主能力优先
关键锚点：
- `packages/coding-agent/README.md`
- `packages/coding-agent/examples/extensions/README.md`
- `packages/coding-agent/src/core/extensions/types.ts`

README 已经明确说：
- 默认不把 sub agents / plan mode 做进 core
- 通过 extensions / packages 去长这些能力

这意味着：

> 高级工作流优先被做成宿主之上的扩展层。

---

## 当前最重要的判断

`pi` 的卖点不是：
- 它多支持几个 provider
- 它有几个 UI 花活
- 它也能做 subagent

而是：

> 它让你不必 fork agent 内核，也能把自己的工作流做成正式 runtime。

这对会自己搭系统的人价值非常高。

---

## 后续要继续验证的点

1. extension runtime 到底覆盖到多深
2. session 树结构是不是宿主路线的真正底座
3. subagent 示例是不是已经足够代表 pi 的官方推荐路线
4. plan mode / provider / UI hostification 到底是“示例级”，还是“平台级”
